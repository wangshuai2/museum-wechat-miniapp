## 一、现状分析

### 1.1 已有功能

| 模块 | 功能 | 完成度 |
|------|------|--------|
| 用户模块 | 登录/注册、个人中心 | ✅ 已完成 |
| 博物馆模块 | 列表展示、详情页 | ✅ 已完成 |
| 展品模块 | 展品浏览、搜索筛选 | ✅ 已完成 |
| 足迹模块 | 参观记录、统计数据 | ✅ 已完成 |
| 预约模块 | 预约管理 | ⚠️ 已有数据模型 |

### 1.2 用户核心痛点

根据需求分析文档，用户面临的主要问题：

1. **记忆模糊** - 去过的博物馆时间长了就忘了
2. **记录分散** - 照片、笔记散落在各处，难以整理
3. **数据丢失** - 换手机、清理相册导致数据丢失
4. **信息不全** - 不知道博物馆有什么、镇馆之宝是什么

### 1.3 功能定位

新功能应聚焦于：
- **强化记录价值**：让参观记录更有意义
- **提升实用价值**：为参观过程提供帮助
- **控制开发成本**：复用现有架构和数据模型

## 功能一：展品笔记系统

### 背景与价值

**用户痛点**：
- 参观时看到感兴趣的展品，想记录知识点但没合适工具
- 拍照后照片散落相册，缺乏关联和整理
- 想回顾参观收获时，找不到对应的记录

**价值主张**：
> "让每一次参观都变成知识积累"

| 价值维度 | 说明 |
|----------|------|
| **记忆留存** | 文字+图片结合，比单纯照片更有记忆价值 |
| **知识沉淀** | 将碎片化的参观体验转化为系统性知识库 |
| **情感连接** | 记录参观时的心情和感受，增强情感记忆 |
| **复用价值** | 笔记可作为攻略分享给其他用户 |

### 用户故事

**故事一：深度参观者**
> 作为一名博物馆爱好者，我希望能在参观时记录展品的背景故事和我的感受，以便回家后能够回顾和整理，让每次参观都有所收获。

**故事二：亲子家长**
> 作为一名带孩子参观的家长，我希望能在参观时记录孩子的提问和我的解答，以便回家后继续延伸学习，培养孩子的兴趣。

**故事三：学生/研究者**
> 作为一名历史专业的学生，我希望能在参观时快速记录展品信息和我的研究想法，以便后续写论文或做研究时使用。

### 功能描述

#### 核心功能

**1. 笔记创建**
- 支持从展品详情页快速创建笔记
- 支持文字+图片混合编辑
- 支持关联展品、博物馆、参观记录
- 支持添加标签分类
- 支持设置隐私级别（公开/仅自己可见）

**2. 笔记管理**
- 按时间线展示笔记列表
- 按博物馆/展品/标签筛选
- 支持搜索笔记内容
- 支持编辑和删除

**3. 笔记展示**
- 笔记详情页支持图文混排
- 显示关联的展品信息卡片
- 支持分享笔记图片（生成精美卡片）

**4. 笔记统计**
- 累计笔记数量
- 笔记字数统计
- 关联展品数
- 常用标签云

#### 交互流程

```
展品详情 → 点击"记笔记" → 进入编辑页 → 输入内容/上传图片 → 保存笔记
                                                                    ↓
                            首页/个人中心 → 我的笔记 → 查看笔记列表 → 点击查看详情
```

#### 数据模型

```typescript
// notes 集合（新增）
{
  _id: ObjectId,
  userId: ObjectId,           // 用户 ID
  exhibitId: ObjectId,        // 展品 ID
  museumId: ObjectId,         // 博物馆 ID（冗余，便于查询）
  visitId: ObjectId,          // 关联参观记录（可选）
  
  title: String,              // 笔记标题（可选）
  content: String,            // 笔记内容（富文本/Markdown）
  images: [String],           // 图片 URL 数组
  tags: [String],             // 标签
  
  visibility: String,         // 可见性：private, public
  likeCount: Number,          // 点赞数（公开笔记）
  
  createdAt: Date,
  updatedAt: Date
}
```

### 验收标准

| 场景 | 预期结果 |
|------|----------|
| 创建笔记 | 用户能从展品详情页创建笔记，支持文字和图片 |
| 保存笔记 | 笔记成功保存，关联展品和博物馆信息 |
| 查看笔记 | 用户能在个人中心查看所有笔记，按时间排序 |
| 编辑笔记 | 用户能修改笔记内容和标签 |
| 删除笔记 | 用户能删除笔记，删除后不可恢复 |
| 搜索笔记 | 用户能按关键词搜索笔记内容 |
| 筛选笔记 | 用户能按博物馆、标签筛选笔记 |
| 分享笔记 | 用户能生成笔记分享图片 |

### 优先级

**P0 - 高优先级**

理由：
1. 直接解决用户核心痛点（记录分散、数据丢失）
2. 与现有足迹模块天然融合，增强产品价值
3. 开发成本可控，主要工作量为前后端开发
4. 为后续社区功能（分享、互动）打基础

### 开发工作量评估

|------|----------|----------|

| 小程序前端 | 笔记编辑页、列表页、详情页 | 3 人天 |
| 测试 | 功能测试、边界测试 | 1 人天 |
| **合计** | | **6 人天** |

## 功能二：收藏与心愿单

### 背景与价值

**用户痛点**：
- 浏览展品时发现感兴趣的，但参观时间有限没看完
- 不知道某个博物馆有什么值得看的，盲目参观
- 想制定参观计划，但缺少工具支持

**价值主张**：
> "把感兴趣的内容先收藏，规划你的专属参观路线"

| 价值维度 | 说明 |
|----------|------|
| **参观规划** | 提前收藏感兴趣的展品，到馆后按清单参观 |
| **持续追踪** | 关注新展览、新展品，及时获取更新 |
| **兴趣发现** | 通过收藏分析，发现用户的兴趣偏好 |
| **社交货币** | 展示自己的收藏品味（公开收藏列表） |

### 用户故事

**故事一：计划型参观者**
> 作为一名喜欢提前做功课的参观者，我希望能在去博物馆前收藏感兴趣的展品，以便到馆后有的放矢地参观，不遗漏精彩内容。

**故事二：回头客**
> 作为一名经常参观博物馆的用户，我希望能在发现喜欢的展品时收藏起来，以便下次来的时候优先参观。

**故事三：展览追踪者**
> 作为一名展览爱好者，我希望能在收藏某个博物馆后收到新展览的通知，以便第一时间知道感兴趣的内容。

### 功能描述

#### 核心功能

**1. 展品收藏**
- 从展品详情页一键收藏/取消收藏
- 支持添加收藏备注（如"下次来看"）
- 显示收藏状态和收藏数量

**2. 博物馆收藏（心愿单）**
- 收藏整个博物馆/展览
- 显示收藏的博物馆列表
- 支持标记参观状态（想去/已去过）

**3. 收藏管理**
- 分 Tab 展示：展品收藏、博物馆收藏
- 支持按博物馆筛选展品收藏
- 支持批量操作（移动、删除）
- 支持拖拽排序（收藏顺序）

**4. 收藏统计与推荐**
- 累计收藏数量统计
- 基于收藏的智能推荐（猜你喜欢）
- 收藏展览更新提醒

#### 交互流程

```
展品详情页 → 点击"收藏"按钮 → 添加到收藏列表（可添加备注）
                                            ↓
                            个人中心 → 我的收藏 → 查看收藏列表 → 点击跳转展品详情
                                            ↓
                            收藏列表 → 按博物馆筛选 → 规划参观路线
```

#### 数据模型

```typescript
// favorites 集合（已有，需扩展）
{
  _id: ObjectId,
  userId: ObjectId,
  
  // 收藏类型
  targetType: String,         // 'exhibit' | 'museum' | 'exhibition'
  targetId: ObjectId,         // 展品ID / 博物馆ID / 展览ID
  
  // 扩展信息
  note: String,               // 收藏备注
  status: String,            // 'want' | 'visited' （想去/已去过）
  sortOrder: Number,          // 排序顺序
  
  // 冗余字段（优化查询）
  targetName: String,         // 目标名称
  targetImage: String,        // 目标图片
  
  createdAt: Date,
  updatedAt: Date
}
```

### 验收标准

| 场景 | 预期结果 |
|------|----------|
| 收藏展品 | 用户能在展品详情页收藏展品，显示收藏状态 |
| 取消收藏 | 用户能取消收藏，收藏列表同步更新 |
| 收藏博物馆 | 用户能在博物馆详情页收藏整个博物馆 |
| 查看收藏 | 用户能在个人中心查看所有收藏内容 |
| 筛选收藏 | 用户能按博物馆筛选展品收藏 |
| 添加备注 | 用户能为收藏添加备注信息 |
| 标记状态 | 用户能标记博物馆为"想去"或"已去过" |
| 收藏数量 | 展品详情页显示收藏数量 |

### 优先级

**P1 - 中高优先级**

理由：
1. 数据库已有 favorites 集合设计，开发成本低
2. 解决用户"不知道看什么"的痛点
3. 增强用户粘性，促进回访
4. 为后续推荐系统和社区功能打基础

### 开发工作量评估

| 模块 | 工作内容 | 预估工时 |
|------|----------|----------|
| 后端 | 扩展 favorites 模型、完善 API | 1.5 人天 |
| 小程序前端 | 收藏按钮、收藏列表页、交互逻辑 | 2.5 人天 |
| 测试 | 功能测试、兼容性测试 | 1 人天 |
| **合计** | | **5 人天** |

## 三、功能对比与推荐

### 对比分析

| 维度 | 展品笔记系统 | 收藏与心愿单 |
|------|-------------|-------------|
| **解决痛点** | 记录分散、数据丢失 | 信息不全、缺乏规划 |
| **用户价值** | 高（知识沉淀） | 中（辅助规划） |
| **开发成本** | 中（6人天） | 低（5人天） |
| **数据模型** | 新增 notes 集合 | 复用 favorites 集合 |
| **技术风险** | 低 | 低 |
| **差异化价值** | 高（竞品少见） | 中（功能较常见） |
| **用户粘性** | 高（内容留存） | 中（功能依赖） |

### 推荐建议

**建议两个功能都纳入开发计划**，理由：

1. **互补性强**：笔记解决"记录"，收藏解决"规划"，形成完整的内容闭环
2. **开发成本可控**：总工作量约 11 人天，可在 2 周内完成
3. **数据模型就绪**：favorites 集合已设计，notes 模型简单清晰
4. **渐进式体验**：用户可以先收藏感兴趣的内容，参观后再补充笔记

**建议开发顺序**：
1. 先开发"收藏与心愿单"（成本低，见效快）
2. 再开发"展品笔记系统"（价值高，增强粘性）
